# 项目规则
不允许乱改我写的代码
禁止私自添加、删除、修改项目中的文件，除非用户特别明确开发范围
禁止删除项目中的文件，除非用户特别明确删除范围
禁止修改我设计好的HTML文件
不需要运行Next服务器，除非这个服务器已经运行了不需要重复启动
我采用的是Tailwind CSS 4.1.18最新版本、Next.js 16.1.1、React 19.2.2版本
- 类型系统 : TypeScript 5
- 代码检查 : ESLint 9
先分析项目架构文件，确定项目的目录结构和文件组织方式，避免重复性创建开发文件
利用现有的架构合理组织项目文件，避免文件冗余和冲突。
- 组件目录 : 所有React组件都应放在`/components`目录下，每个组件有一个单独的文件。
- 页面目录 : 所有Next.js页面都应放在`/pages`目录下，每个页面有一个单独的文件。
- 样式目录 : 所有Tailwind CSS样式都应放在`/styles`目录下，每个样式文件有一个单独的文件。
- 公共资源目录 : 所有公共资源（如图片、字体等）都应放在`/public`目录下。
- 其他目录 : 其他项目相关目录（如`/src`、`/app`等）根据需要组织。
在 React 中，HTML 属性需要遵循 camelCase 命名规范，而不是使用短横线命名法。例如，`class` 应写为 `className`，`for` 应写为 `htmlFor`。
# 必须遵守
1. **不假设原则**：不假设任何自定义组件或文件存在
2. **询问优先**：对不确定的引用先询问或提供选项
3. **方案提供**：为可能缺失的文件提供创建代码
4. **路径安全**：优先使用相对路径，其次考虑别名
# 【JSX安全规则】
1. 所有JSX元素必须正确闭合
2. 条件渲染必须用括号包裹
3. 注释必须使用 {/* 注释内容 */}
4. 属性值必须用引号或花括号
5. 列表项必须提供key属性
# 【错误预防措施】
1. 避免在JSX中使用//注释
2. 确保三元运算符?:的三个部分都存在
3. 检查所有<div>都有对应的</div>
4. 确保函数返回单个JSX元素（或Fragment）
# 【代码结构要求】
1. 'use client' 指令
2. 导入语句（分组：react，第三方库，类型）
3. 类型/接口定义
4. 组件函数定义（包含props解构和类型）
5. 状态声明（useState, useRef等）
6. 处理函数定义
7. 返回JSX（确保所有标签闭合）
# 关键要求：
1. 所有样式类名必须与原型HTML完全一致
2. 所有交互功能必须完整实现
3. 代码必须是可运行的，无任何语法错误
4. 组件拆分解耦，遵循React最佳实践
5. 完全响应式设计，与原型一致
